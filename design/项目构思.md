# AFSA — AI-Driven Fluid Software Architecture
**通用框架设计方案 v0.1**

---

## 1. 背景与动机

传统软件存在一个根本矛盾：用户需求是千变万化的，但软件迭代周期是缓慢的。现有 AI 编程工具（如 Cursor、GitHub Copilot）服务的是开发者，而非最终用户。

AFSA 的核心思想是：**将"需求 → 开发 → 交付"这个循环内嵌进软件本身**。每个软件自带一支 AI 驱动的开发团队，能够根据用户的自然语言指令进行实时个性化改造，同时保证安全边界与系统稳定性。

---

## 2. 框架总览

AFSA 由四个层级构成，各层职责清晰、可独立替换：

| 层级 | 名称 | 核心职责 |
|------|------|----------|
| L1 | AI Team Layer | 理解用户需求，拆解并执行开发任务 |
| L2 | Orchestration Layer | 任务调度、沙箱验证、版本管理 |
| L3 | Business Implementation Layer | 业务逻辑实现，分可变区与不变核心 |
| L4 | Safety & Governance Layer | 权限控制、审计日志、一键回滚 |

---

## 3. 各层详细设计

### L1 — AI Team Layer（可插拔层）

本层**完全可外包给现成开源框架**（如 CrewAI、AutoGen、LangGraph），无需自研。内部固定三类角色：

**PM Agent**
唯一的用户入口。用自然语言与用户沟通，识别并澄清模糊需求，将其结构化为可执行任务卡。不涉及任何代码细节，是用户与技术世界之间唯一的翻译层。

**Architect Agent**
内部技术把关者。在 Dev Agent 动手前，评估改动是否会破坏系统整体结构，判断是否在安全边界内，输出技术可行性报告。

**Dev Agents**
专业执行者，按职能分工：
- Frontend Agent — UI / 交互层改动
- Backend Agent — 服务 / 业务逻辑改动
- Data Agent — 数据结构改动

---

### L2 — Orchestration Layer（编排层）

AI Team 与业务代码之间的桥梁，负责任务的完整生命周期管理：

**Task Dispatcher** — 任务队列管理与并发控制，追踪每个任务的执行状态。

**Sandbox Runner** — 所有代码变更必须先在隔离沙箱中验证通过，才能进入合并流程。这是防止错误变更进入生产环境的关键屏障。

**Version Control** — 每次变更均保存完整快照，为 Rollback Engine 提供基础数据。

---

### L3 — Business Implementation Layer（业务层）

核心设计原则：**双区隔离**。

| 区域 | 内容 | Agent 权限 |
|------|------|------------|
| Mutable Zone（可变区） | 插件、配置项、UI 模块、个性化规则 | 可读可写 |
| Immutable Kernel（不变核心） | 核心认证、数据 Schema、关键业务规则、安全策略 | 只读 |

这道边界保障了：无论 Agent 如何操作可变区，系统的核心稳定性与数据完整性不受影响。

---

### L4 — Safety & Governance Layer（治理层）

横切所有层级的安全兜底机制：

**Permission Guard** — 每个任务执行前进行权限验证，可按用户角色或租户差异化配置。

**Audit Trail** — 完整记录每一次 Agent 操作的日志，包括操作人、时间、变更内容、执行结果。

**Rollback Engine** — 基于 Version Control 快照，在任意时刻将系统回退至指定的稳定版本，支持部分回滚与全量回滚。

---

## 4. 端到端协作流程

```
用户
 │  自然语言需求
 ▼
PM Agent ──→ 需求澄清 & 结构化任务卡
 │
 ▼
Architect Agent ──→ 技术可行性评估 & 安全边界审查
 │
 ▼
Task Dispatcher ──→ 分发至对应 Dev Agent
 │
 ▼
Dev Agents ──→ 在 Mutable Zone 内实施变更
 │
 ▼
Sandbox Runner ──→ 隔离验证
 │
 ▼
Permission Guard ──→ 最终权限核查
 │
 ▼
变更合并 + Audit Trail 记录 + Version Control 快照
 │
 ▼
PM Agent ──→ 向用户确认结果（自然语言）
```

---

## 5. 框架通用性

**AI Team 层完全可替换**
本层通过标准接口（任务卡格式）与 Orchestration 层通信。更换底层 Agent 框架只需替换 L1，其余三层完全不受影响。

**双区设计与具体业务无关**
Mutable Zone / Immutable Kernel 的划分模式适用于 SaaS 产品、移动 App、企业内部系统等任何软件形态。各项目只需根据自身情况定义哪些属于"核心"即可。

- PM Agent 的需求澄清对话流程设计
- 开源框架接入规范（接口适配层设计）
